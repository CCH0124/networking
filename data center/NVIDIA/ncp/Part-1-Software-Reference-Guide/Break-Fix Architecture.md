# Break-Fix Architecture

NCP 軟體參考架構中故障修復 (Break-Fix) 系統的任務，是透過自動化確保租戶在叢集上擁有高可用性，這些自動化包括：

* 確保雲端供應商向租戶交付健康的叢集
* 被動篩選、分類並修復硬體（現場診斷 field diags ➔ 換貨授權 RMA）和軟體元件（應用韌體更新、軟體升級與系統重新啟動）
* 盡可能減少修復損壞節點所需的人工干預

其重點在於建立一個專注於監控的 Day 2 故障修復系統，並在最大程度上將基於 GPU 的運算節點之修復工作自動化。Day 2 定義為資料中心上線、經過驗證並準備好進行正常租戶營運之後。在此階段，將停機時間降至最低是至關重要的。

## 故障修復概述 (Break-Fix Overview)

故障修復架構的意圖在於對 AI 基礎設施的 GPU 節點進行「偵測、分類、修復並驗證」。這有時被稱為健康檢查與故障修復，本文件將故障修復視為包含健康檢查在內的整體流程。本節的目標在於描述如何建立一個強大的故障修復架構，並使其能夠在廣泛的 AI 平台和工作負載中運行，且由 NCP 來營運。

雖然該架構可以擴充以支援傳統的運算節點，但本文件的重點在於 GPU 節點。透過建立自動化處理故障修復場景的方法，特別是那些完全不需要人工干預即可佈署完整修復的場景，可以顯著提升系統的運作時間 (Uptime)。

## 故障修復 (Break-Fix)

故障修復架構應被視為包含兩個主要領域。第一個領域是**租戶領域 (Tenant Domain, TD)**，包括在租戶擁有該節點或 GPU 時進行的所有故障修復操作。第二個領域是**基礎設施營運商領域 (Infrastructure Operator Domain, IOD)**，包括在機器處於基礎設施營運商控制之下時進行的所有故障修復操作。此劃分的一個顯著例外是，即使機器屬於租戶領域，IOD 仍會執行持續的/離線的檢查。將這兩個領域在邏輯上保持分離是非常重要的。

租戶領域 (TD) 有三個主要角色：

* **持續的頻內 (In-band) 節點健康檢查**：這需要租戶以定期訊號 (Heartbeat) 執行一組健康檢查，向故障修復系統通知每個節點的整體健康狀況。因此，租戶必須選擇加入 (Opt-in) 執行持續健康檢查，並參與整體的故障修復流程。
* **租戶端修復 (Tenant Remediation)**：在可能的情況下，故障修復系統的目標是在盡量減少對租戶影響的前提下，以最快速度修復任何不健康的節點。許多問題可以在不將節點移出循環的情況下進行處理，例如執行 PCIe GPU 重設 (Reset)。
* **節點歸還 (Node Return)**：如果節點的健康檢查識別出一個無法在租戶擁有期間修復的問題，租戶必須同意將節點歸還給 IOD 領域以進行進一步操作。這可能會導致為租戶分配另一個節點，或者可能導致租戶可用的能力降低。

基礎設施營運商領域 (IOD) 有三個主要角色：

* **初始節點驗證**：一個關鍵原則是確保 IaaS 服務在最初提供時，始終向租戶交付一個已知的正常 (Known-good) 節點。因此，當節點處於 IOD 控制下時，基礎設施營運商需要進行節點健康檢查，並確保節點如預期般運作（在測試所能顯示的範圍內）。
* **持續的離線節點健康檢查**：某些健康檢查可以以離線方式進行，因此不需要租戶的選擇加入。平台或資料湖上的離線檢查範例包括：針對節點溫度的 BMC 查詢、DPU 心跳訊號監控，以及用以尋找問題跡象的日誌檔案監控。如前所述，即使機器在租戶領域中，此檢查仍由 IOD 執行。
* **節點修復**：當透過任何機制識別出節點問題時，以儘可能自動化的方式處理該節點，將該節點移出循環、使其恢復健康，然後重新投入循環中。

### 節點級健康檢查 (Node-Level Health Checks)

不論是由 NVIDIA 建立的工具還是生態系統中的工具，都有許多公用程式可以協助進行節點級健康檢查。對於 NCP 軟體參考指南架構，不假設特定的工具整合，但假設使用了 NVIDIA 的資料中心 GPU 管理器 (Data Center GPU Manager, DCGM) 實體。

健康檢查可以在基礎設施的所有層級運行。這是因為某些資訊僅在堆疊的特定層級可用。例如，許多元件需要直接存取 GPU 驅動程式。這可能存在於主機作業系統 (Host OS) 層級，或者可能作為傳遞 (Passthrough) 裝置提供給虛擬機 (VM) 層級。其結果是，特定的一組健康檢查需要在適當的位置運行。

健康檢查實體通常包括一個測試框架 (Harness) —— 用於以定期的心跳訊號運行本機健康檢查，並與整體的故障修復解決方案進行互動 —— 以及檢查項目本身。某些健康檢查可能在每個時間間隔運行，而其他檢查可能運行頻率較低，或僅在收到請求時運行。當檢測到健康問題時，健康檢查實體將向故障修復控制平面報告以採取行動。

## 重點整理 (Key Takeaways)

1. **故障修復 (Break-Fix) 的 Day 2 目標**
   * 在資料中心上線並驗證後的正常租戶營運階段 (Day 2)，故障修復系統旨在藉由「偵測、分類、修復與驗證」流程，最大程度自動化 GPU 運算節點的維護，減少人工干預，從而將停機時間降至最低。

2. **租戶領域 (TD) 與營運商領域 (IOD) 的雙域邏輯分離**
   * **租戶領域 (TD)**：處理租戶持有節點時的維護。租戶需選擇加入以運行**頻內節點健康檢查**。在可能情況下由系統在不中斷租戶前提下進行**租戶端修復**（例如 PCIe GPU 重設）；若發生重大硬體損壞，則進行**節點歸還**，將節點送回 IOD 處理。
   * **營運商領域 (IOD)**：處理機器在營運商控制時的維護。負責提供時的**初始節點驗證**（交付已知正常節點），並執行**持續離線節點檢查**（即使在租戶持有期，亦由 IOD 透過 BMC 溫度查詢、DPU 心跳或日誌監控來執行），並在發現問題時進行**自動化節點修復**（移出循環、修復後再重新投入）。

3. **節點級健康檢查之分層實作**
   * 由於特定資訊的獲取取決於堆疊層級（有些需直接存取 Host OS，有些可透過 Passthrough 至 VM 層），因此健康檢查會分層在最適當的位置執行。
   * 架構以 **NVIDIA DCGM (資料中心 GPU 管理器)** 作為健康檢測之核心，並藉由測試框架 (Harness) 定期向故障修復控制平面回報，以採取相對應的自動化處置。
