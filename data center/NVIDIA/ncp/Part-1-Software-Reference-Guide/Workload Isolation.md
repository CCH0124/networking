# Workload Isolation

在多租戶部署中，端到端解決方案提供工作負載隔離與韌性是至關重要的。在多租戶 K8s 部署中，終端使用者應該能夠請求專屬的 K8s 控制平面實例 (KaaS)。為租戶提供 API 存取的 K8s 叢集實例在租戶之間**絕對不能共享**，以確保隔離性與安全性。然而，在許多部署場景中，運行控制平面節點虛擬機的「實體工作主機」可以在租戶之間共享，這是提高底層實體伺服器利用率的極佳方式。

從租戶的角度來看，裸機 (Bare Metal) 透過消除吵鬧鄰居 (Noisy neighbor) 效應並顯著減少基礎設施營運商可能擁有的存取權限，提供了最佳的節點級隔離，但其成本最高（用於基礎設施、佈署、節點回收期間的清理消毒等）。因此，虛擬機 (VM) 很可能以較低成本滿足大多數的運算需求。

工作節點是運行在實體工作主機上的虛擬機 (VM)。託管這些 VM 的超級監督器 (Hypervisor) 應將其進行隔離，並保護其免受主動和被動攻擊，包括資源競爭攻擊、記憶體洩漏、CPU 快取行 (Cache line) 洩漏等。NVIDIA 建議在超級監督器中配置防護措施，以防止敵對或吵鬧的鄰居干擾 GPU、CPU、記憶體和網路等實體資源，以及儲存和 IP 空間等邏輯資源。

單一節點上的多租戶（即來自不同租戶的虛擬機在同一個實體節點上執行）本質上比僅有單一租戶存在時更不安全。在可能的情況下，建議將節點保持對齊至單一租戶，因為這能提供最佳的安全性，並能使附加的 GPU 盡可能高效地協同工作。如果將多個租戶分配到同一個節點，則必須擁有強大的隔離機制（例如：硬化的超級監督器、使用 DPU）。

對於託管式 Kubernetes 和 AI 平台等使用場景，控制平面側的資源也應該按租戶進行分配，以確保喧鬧或行為異常的租戶不會妨礙另一個租戶的運作。具體而言，對於像 K8s 控制平面這樣的服務，建議為每個租戶運行一個獨特的控制平面。如果隔離解決方案足夠強大，將這些控制平面聚合在運行於相同機器上的虛擬機中是可以接受的。

當基礎設施在租戶之間移交時，營運商必須從基礎設施中清除所有租戶資料（即進行「清理消毒 sanitize」）。這包括擦除記憶體、確保所有裝置的韌體均處於已知的正常狀態，以及「擦除」本機暫存硬碟的資料。

一個工作節點（在實體工作主機上實例化的工作虛擬機）必須分配給恰好一個租戶，因此也只能分配給一個 K8s 控制平面實例。

當虛擬機被作為工作節點分配給 K8s 控制平面實例時，控制平面可以一次在該虛擬機中調度一個或多個工作負載。由 runc 或類似的基於程序 (Process-based) 的隔離所提供的虛擬機內部工作負載隔離，可能不如超級監督器在虛擬機之間提供的隔離強大。營運商和租戶在其中調度工作負載時，應考慮到這些使用者對使用者 (User-to-user) 隔離的差異。K8s 控制平面應支援租戶選擇在每個工作節點虛擬機上調度一個或多個工作負載。

應主動監控主機上的工作節點，以發現資源競爭、濫用和攻擊模式。對於表現出異常行為的工作虛擬機，應採取適當的行動。

有關網路基礎設施中的工作負載隔離，可在《通用網路參考架構》(Common Networking reference architecture, ID 1144496) 中取得。請向您的 NVIDIA 銷售代表請求此文件。

## 啟動與證明 (Boot and Attestation)

對運算節點上運行的韌體和軟體的完整性 (Integrity) 充滿信心是至關重要的。這是建立安全基礎設施的基石。雲端營運商應對基礎設施採取零信任 (Zero-trust) 方法，並實作能夠證明其狀態的安全且受度量啟動 (Measured boot) 解決方案。此流程應應用於資料中心內運行作業系統的每個處理器。這包括 BMC、DPU 和主機處理器（x86 或 ARM）。安全的多租戶 AI 基礎設施應實作安全啟動、清理與證明方法。

### 安全啟動 (Secure Boot)

安全啟動是保證載入的特權軟體已被信任的供應商或營運商簽章的程序。這是透過在一個經驗證的軟體實體與下一個實體之間建立信任鏈來實現的。

![安全啟動階段](https://files.buildwithfern.com/nvidia-dsx.docs.buildwithfern.com/dsx/a9c92a2573242df97038e0fdf111c93add5b976f299c6b17f0200cbcf5a9cd51/_dot_dot_/docs/ncp/software-reference-guide/assets/images/ncp-srg-secure-boot-stages.png)

這個鏈條可以根據營運商的意願持續向堆疊上方擴充。在此程序結束時，只有被允許的代理程式簽章的受信任程式碼才會被載入。

### 受度量啟動 (Measured Boot)

受度量啟動旨在提供一條路徑，用以檢測安全啟動策略或簽章金鑰遭到破解的情況。因此，它不能防止壞事發生，而是提供已載入並運行之內容的審計軌跡 (Audit trail)。這是藉由在上述每個階段不僅驗證簽章，而且生成正在載入的整個二進位映像檔的雜湊值（稱為摘要 digest），並將該摘要儲存（擴充）到稱為 TPM（可信賴平台模組, Trusted Platform Module）的平台元件中來實現的。這條軌跡現在允許控制平面能夠驗證處理器是否載入了預期的軟體順序。

### 遠端證明 (Remote Attestation)

驗證載入了正確軟體的程序稱為遠端證明。作為雲端控制平面的一部分，證明服務 (Attestation service) 應向每個主機的 TPM 請求當前的度量值。這包括稱為 Nonce 的挑戰值，以確保回應不會被欺騙。在不深入討論詳細密碼學握手的情況下，TPM 會回應證明服務，證明服務作為驗證器來確認 TPM 的回應是否正確，且所有度量均使用適當的金鑰和憑證進行，並與黃金標準值 (Golden values) 一致。

## 重點整理 (Key Takeaways)

1. **多租戶環境下的工作負載隔離原則**
   * **控制平面安全**：租戶之間的 K8s 叢集控制平面實例**絕對不能共享**，以確保 API 存取的安全性。實體工作主機則可以在租戶之間共享以提升底層物理伺服器的利用率。
   * **節點唯一性**：每個工作虛擬機 (Worker VM) 必須且只能分配給單一租戶與單一 K8s 控制平面。
   * **共享主機防護**：若多租戶的 VM 運行於同台實體節點上，需配置超級監督器防護，避免資源競爭攻擊、記憶體洩漏與 CPU 快取行 (Cache line) 洩漏，並結合 DPU 進行硬體級隔離。
   * **VM 內部隔離警告**：VM 內部的容器程序級隔離 (runc) 弱於 VM 間的 hypervisor 隔離，調度時需留意 user-to-user 安全性。

2. **基礎設施移交時的清理消毒 (Sanitization)**
   * 在資源回收並分配給新租戶時，營運商必須執行徹底的清理消毒：擦除記憶體、將所有設備的韌體回復到已知正常狀態，並抹除本機暫存硬碟 (Local Ephemeral) 的數據。

3. **零信任安全底座：安全啟動與證明**
   * **安全啟動 (Secure Boot)**：建立簽章信任鏈，確保只載入被允許的代理程式簽章的受信任代碼。
   * **受度量啟動 (Measured Boot)**：不只驗證簽章，還將各階段載入的映像檔雜湊摘要記錄（擴充）至 TPM 晶片中，形成不可篡改的軟體載入審計軌跡。
   * **遠端證明 (Remote Attestation)**：控制平面的證明服務向主機 TPM 發起加密挑戰 (Nonce)，確認 TPM 回應的度量值與黃金標準值 (Golden values) 相符，以驗證韌體與軟體完整性。
