# Networking Integrations (網路整合)

NVIDIA Infra Controller (NICo) 與各種網路虛擬化解決方案整合，允許租戶的裸金屬執行個體（Instances）在隔離的分區上進行通訊。任何不屬於同一個分區的執行個體都無法參與通訊 —— 無論這些執行個體是屬於同一個租戶還是不同的租戶。

NICo 中的網路整合透過以下模式來實現此目的：

---

## 工作流程 (Workflows)

### 租戶分區管理 (Tenant Partition Management)

1.  租戶擁有可用於管理其執行個體的一組網路分區之 API。這些分區的範例如下：
    *   **VPC**（用於乙太網路）
    *   **InfiniBand 分區**
    *   **NVLink 邏輯分區**
2.  可能會提供額外的子 API 以進行更深入的分區管理，例如需要動態將資源（如 IP）新增至分區中。
3.  租戶可以透過 API 查詢分區的狀態。每個分區都有一個生命週期狀態（`Provisioning`（建置中）、`Ready`（就緒）、`Terminating`（終止中））。
4.  只有當不再有與之相關聯的執行個體時，分區才能被完全刪除。針對這些物件的狀態機透過檢查終止中狀態來確保這一點。
5.  管理員工具（網頁介面和 admin-cli）使站點管理員能夠瞭解這些資源及其狀態。

### 租戶執行個體介面組態設定 (Tenant Instance Interface Configurations)

1.  租戶能夠將其執行個體的網路介面，與其預先建立的分區進行關聯。此組態設定可以在建立執行個體時進行，也可以在稍後使用 `UpdateInstanceConfig` 呼叫來進行。
2.  為了支援在執行個體之上運行虛擬機器（VM），分區應該是基於「每個介面（per-interface）」而非基於「每台主機（per-host）」進行設定的。這允許 VM 系統將不同的介面（PCI PF）附接到不同的 VM 上。
3.  當執行個體被更新時，如果機器上的網路已使用屬性中作為執行個體狀態一部分的 `configs_synced` 重新設定為使用新分區，租戶將會獲得精確的狀態。此旗標還將影響在 `state` 欄位中顯示的執行個體整體就緒狀態：如果網路尚未完全設定，執行個體將顯示 `Configuring`（設定中）狀態。一旦網路設定完成，它將轉移至 `Ready`（就緒）狀態。
4.  更新執行個體組態設定時，作為 `Instance` 一部分的 `config_version` 欄位會遞增。
5.  在初始建置時，狀態機會阻止開機進入租戶 OS，直到達到期望的組態設定為止。這保證了一旦執行個體開機，它便能立即與共享該分區的租戶之所有其他執行個體進行通訊。
6.  在執行個體終止時，終止流程會阻塞（Block），直到網路介面被重新設定為不再屬於任何分區為止（該執行個體已在網路上被隔離）。這確保了一旦租戶收到執行個體已被刪除的通知，它至少已經被完全隔離，並且不會再顯示為「幽靈執行個體（Ghost instance）」——即使硬碟可能尚未清理乾淨。在此工作流程中，由租戶提交並反映在 `InstanceConfig` 訊息中的「期望」執行個體組態設定不會變更。這意味著系統還必須考慮機器物件中的另一個欄位，以便從「租戶期望的網路」切換到「隔離網路」。

### 機器功能與執行個體類型 (Machine Capabilities and Instance Types)

*   租戶需要知道他們實際上該如何設定其執行個體。合法的組態設定取決於硬體。例如，在一個配備 4 個已連接 InfiniBand 連接埠的執行個體中，租戶可以將這些連接埠的每一個與不同的分區相關聯。然而，租戶無法為沒有 InfiniBand 連接埠的執行個體設定 IB。
*   租戶透過包含功能清單的「執行個體類型（Instance Types）」來瞭解支援的組態設定。每種網路功能類型都會告知租戶如何設定對應的介面。這意味著對於每個可設定的介面，執行個體類型都應列出相應的功能。
*   編碼在執行個體類型中的功能集合，必須與 `Machine` 相關聯的功能集合相符或為其子集。機器功能是在硬體發現和接入（Ingestion）階段偵測到的。站點管理員可以透過除錯工具查看這些功能。
    *   在機器接入期間，會透過頻內（使用 scout）和頻外（使用 site-explorer）收集有關所有網路介面的資料。該資料儲存在 `machine` 和 `machine_topologies` 資料庫資料表中。
    *   基於原始發現資料，核心服務會計算出「機器功能（Machine Capabilities）」（類型為 `MachineCapabilitiesSet`）並呈現給站點管理員。這些功能告知使用者可設定的介面數量。對於每種網路整合，都需要一種新型的機器功能。例如，InfiniBand 使用 `MachineCapabilityAttributesInfiniband` 功能，而 NVLink 使用 `MachineCapabilityAttributesGpu` 功能。
*   SKU 驗證（SKU Validation）功能可以包含檢查任何新接入的主機是否包含預期數量的網路介面 —— 其中每個網路介面通常被描述為一種機器功能。

## 實現需求與考量 (Implementation Requirements and Considerations)

為了實現這些工作流程，在 NICo 中開發並驗證了以下成功模式：

### 網路介面的期望狀態與實際狀態 (Desired State vs Actual State of Network Interfaces)

*   對於每台機器上的每個網路介面，NICo 都會追蹤期望狀態（目標網路分區和其他組態設定）以及實際狀態。
*   期望狀態是「租戶請求的狀態」與由 NICo 內部管理的組態設定集合的結合。
    *   租戶請求的狀態完全儲存在 `InstanceConfig` 物件中。
    *   內部請求的狀態儲存在 `ManagedHostNetworkConfig` 中，該組態是資料庫 `machine` 資料表的一部分。這裡最關鍵的欄位是 `use_admin_network` 欄位，它控制是否覆蓋租戶組態設定，以及是否確實應將機器放置在隔離/管理網路上。
*   實際狀態儲存為 `Machine` 資料庫物件的一部分。NICo 與相應網路子系統之間的整合負責在該處更新它。NICo 內部的所有其他工作流程都將使用此觀測到的狀態進行決策，而不會去調用任何外部服務。這種內部快取觀測狀態的方法使工作流程保持確定性與可靠性，因為它們基於同一個單一真實水源（Source of Truth）進行操作。它也有助於提高反應能力和擴充性，因為所有其他程式碼路徑都不再需要調用外部服務來瞭解網路狀態。

    隨著時間推移，這裡開發了兩種整合模式：
    1.  **監控與調和任務模式**：實際觀測到的狀態由特定於該網路技術的「監控與調和任務」來更新。此類整合的範例包括 `IbFabricMonitor` 服務（用於 InfiniBand）與 `NvlPartitionMonitor`（用於 NVLink）。如果外部網路是透過外部服務控制的，則這種監控和整合方式最為有利，因為整合程式能夠同時獲取多個裝置和主機的實際網路狀態，並可以一次更新所有受影響的機器物件。
    2.  **API 主動回報模式**：由與該介面相關聯的服務透過向 NICo 發起 API 呼叫，為每個介面或主機更新其實際觀測到的狀態。此類整合的範例是 `dpu-agent` 透過 gRPC 呼叫（`RecordDpuNetworkStatus`）發送觀測到的 DPU 組態設定。
*   站點管理員需要能夠查看任何介面的期望組態設定以及實際組態設定。

### 狀態調和 (State Reconciliation)

需要有一種機制來定期比較預期的網路組態設定與期望的網路組態設定。如果它們不同步，相應的組件需要採取所有必要的操作來使組態設定恢復同步。

1.  對於使用外部服務來控制分區的網路技術（NVLink、InfiniBand），可以使用 `Monitor` 背景任務來達到此目標。如果它們偵測到組態設定不匹配，就會向外部網路服務執行 API 呼叫以解決問題。
2.  對於其他整合，外部代理程式可以拉取任何主機的期望組態設定，執行（可能是本機的）組態設定變更，然後將新狀態報告回 NICo。DPU 即採用此種方法。

### 執行個體生命週期與「租戶回饋」 (Instance Lifecycle and "Tenant Feedback")

1.  `InstanceStatus` 應定義一個 `configs_synced` 欄位，用以顯示該執行個體所有介面的網路組態設定是否已套用。除了整體的 `configs_synced` 值之外，每種網路整合也應該有一個對應的 `configs_synced` 欄位（例如 `InstanceStatus::infiniband::configs_synced`）。
    *   每種技術專屬的 `configs_synced` 欄位值，應透過將期望的網路組態設定與儲存在 `Machine` 物件中的實際組態設定進行比較來導出。這是在 `InstanceStatus::from_config_and_observation` 內實現的。
    *   彙總的 `configs_synced` 欄位值是 `InstanceStatus` 訊息中所有各別 `configs_synced` 欄位做邏輯與（AND）的結果。
2.  執行個體的租戶狀態（透過 `Instance::status::tenant::state` 進行通訊）應考慮期望的組態設定是否已套用：
    *   如果執行個體仍處於建置狀態之一（`Ready` 之前的任何狀態），它將顯示租戶狀態 `Provisioning`（建置中）。
    *   如果執行個體曾處於 `Ready` 狀態，且實際網路組態設定偏離了期望的組態設定，狀態應顯示為 `Configuring`（設定中）。
    *   如果已請求終止執行個體，執行個體的狀態應顯示為 `Terminating`（終止中），與網路組態設定無關。
3.  執行個體狀態機應該在某些狀態中設置保護條件（Guards），以等待期望的網路組態設定被套用：
    *   在初始執行個體建置期間（進入 `Ready` 狀態之前），狀態機中的一個狀態應等待期望的網路組態設定套用。對於 DPU 組態設定，這發生在 `WaitingForNetworkConfig` 狀態中。此狀態下的保護條件應使用與導出租戶的 `configs_synced` 值相同的邏輯。
    *   在執行個體終止期間，狀態機中的一個狀態應等待機器與網路中的任何其他機器隔離。如果省略了此步驟（以允許機器在網路結構不健康的情況下繼續終止），則相應的機器必須至少被標記健康警報，以防止其他租戶使用該主機。這兩個選項都保證了其他租戶不會獲得對該租戶網路分區的存取權。

### 機器功能與執行個體類型擴充 (Machine Capabilities and Instance Types)

1.  對於每一種新型網路技術，都需要擴充機器功能的定義。
2.  硬體列舉程序需要更新，以便獲取並儲存新類型的功能。

### 網狀架構健康狀態監控與除錯功能 (Fabric Health Monitoring and Debug Capabilities)

1.  如果某個網路子系統是透過外部網狀架構監控服務進行管理的，則應監控該服務的健康狀態（如 NICo 所見），以便允許 NICo 管理員瞭解是否存在會導致網路組態設定無法套用的上游問題。應該監控的常見指標包括上游服務可用性（請求成功率）以及任何 API 呼叫的延遲。
2.  對於某些網路技術，NICo 整合了除錯工具，允許 NICo 操作員在不需要憑證的情況下查看網狀架構管理服務的狀態。NICo 中的 UFM 探測器（UFM Explorer）功能就是此類工具的一個範例。對於任何未來的整合，若有可能也應整合類似的工具。

## 可組態性 (Configurability)

*   在 NICo 部署中，某個網路虛擬化技術是否可用，應該能夠透過 NICo 組態設定檔進行設定。

---

## 受控主機強制刪除支援 (Managed Host Force Delete Support)

*   當主機從系統中被強制刪除時，它不會經歷常規的撤銷建置（Deprovisioning）狀態。這意味著如果沒有額外的支援，該主機的網路組態設定仍會保留在外部代理程式和網狀架構管理器中。
*   為了防止這種情況，強制刪除程式碼路徑應包含額外的邏輯，以透過外部網狀架構管理器 API 將主機自各分區中分離。

---

## 外部網狀架構管理器用戶端程式庫 (External Fabric Manager Client Libraries)

*   如果使用外部網狀架構管理器來觀測介面狀態並設定配置，則需要一個 Rust 版本的用戶端程式庫。
*   與外部網狀架構管理器的互動將需要憑證。這些憑證應從檔案系統中讀取，並透過外部服務（例如 K8S Secret）注入。

---

## 重點整理

1.  **多維度網路分區管理**：NICo 支援 VPC（乙太網路）、InfiniBand 分區與 NVLink 邏輯分區等多種網路虛擬化技術，以確保不同分區的裸金屬執行個體在物理/邏輯上徹底隔離，非同分區實例無法進行任何通訊。
2.  **基於介面（Per-Interface）的細粒度配置**：為了支援在裸金屬實例上運行虛擬機器（VM），網路分區是基於「每個網路介面」而非「每台主機」進行設定。這使 VM 監控系統能將各別的實體介面（PCI PFs）分派給不同 VM。
3.  **快取實際狀態以實現確定性調和**：
    *   **期望狀態**：由租戶的 `InstanceConfig` 與 NICo 內部的 `ManagedHostNetworkConfig`（如 `use_admin_network` 欄位）共同決定。
    *   **實際狀態**：儲存在 `Machine` 物件中。NICo 所有背景工作流程都僅依賴此內部快取狀態做決策，不調用外部服務，保證狀態判斷的確定性與效能。
    *   **兩類更新模式**：外部管理時使用監控調和任務定期抓取（如 `IbFabricMonitor`, `NvlPartitionMonitor`）；非外部管理則由 Agent 發起 API 主動回報（如 DPU 代理程式呼叫 `RecordDpuNetworkStatus`）。
4.  **生命週期防禦保護條件（Guards）**：
    *   **開機保護**：初始建置時，狀態機會在 `WaitingForNetworkConfig` 狀態下阻塞開機，直至期望的網路配置完成，確保主機一旦引導即可直接連通。
    *   **終止隔離**：終止實例時，狀態機阻塞流程直至介面完全登出所有分區；即使遭遇不健康的 Fabric 結構，也必須在強制清除前對機器標記健康警報，防範下一個租戶意外連入舊有的網路分區。
5.  **強制刪除邏輯補強**：當主機被強制刪除時（跳過常規生命週期），NICo 會執行專屬邏輯，主動向外部 Fabric 管理器（如 UFM 或 NMX-M）發起 API 呼叫，強行解除該主機與各分區的綁定，避免配置殘留。
