# InfiniBand Connections

## InfiniBand Architecture

### 1. 實體層 (Physical Layer)

* **核心功能：** 負責在實體線路上傳送位元 (Bits)，定義傳輸速率（如 HDR 200Gbps、NDR 400Gbps、XDR 800Gbps）並確保訊號的完整性。這就像是高速公路的實體路面與基礎設施。
* **對應硬體：** 銅纜 (DAC)、光纖 (AOC)、連接口 (QSFP, OSFP)，以及光電轉換收發器。

### 2. 連結層 (Link Layer)

* **核心功能：** 確保兩個節點間（Hop-to-hop）的可靠傳輸，處理封包框架 (Packet framing) 與 CRC 錯誤偵測。它最核心的機制是透過**信用額度機制 (Credit-based system) 來防止封包遺失**，並支援多個虛擬通道 (Virtual Lanes, VLs) 實現 QoS 控制。這相當於道路上的車道管制與交通規則。
* **對應硬體：** 伺服器端的主機通道介面卡 (Host Channel Adapter, HCA) 的資料層 (Data Plane)，以及 InfiniBand 交換器底層的 ASIC 晶片。

### 3. 網路層 (Network Layer)

* **核心功能：** 決定資料在 InfiniBand 子網路 (Subnet) 內的傳輸路徑，以提供極低延遲且具備決定性 (Deterministic) 的路由。它完全依賴集中式的控制機制，不使用傳統廣播。這層就像是導航與交通控制系統。
* **對應硬/軟體：** 主要依賴**子網路管理員 (Subnet Manager, SM)** 這套控制層 (Control Plane) 軟體來運作。SM 會分配區域識別碼 (LID) 作為定址基礎，並將路由表 (Routing tables) 寫入交換器硬體中。

### 4. 傳輸層 (Transport Layer)

* **核心功能：** 負責端到端 (End-to-end) 的高效資料傳輸，執行 RDMA 操作，達成**零複製 (Zero-copy) 傳輸**，使資料移動完全不需 CPU 介入。這層可類比為快遞服務的運送方式（特急、標準等）。
* **對應硬/軟體：** 定義了不同的傳輸類型（如 RC 可靠連線、UC 不可靠連線、UD 不可靠資料報）與收發佇列 (Queue Pairs, QP)。這些傳輸邏輯與連線處理皆由 HCA 網路卡*硬體*直接負責，而非依賴作業系統軟體。

### 5. 上層應用層 (Upper Layer)

* **核心功能：** 負責執行實際的運算工作負載與應用程式介接，是大規模分散式運算與 AI/ML 訓練叢集的核心。
* **對應硬/軟體：** 包含各類伺服器、HPC 運算節點、AI 框架與儲存系統。在軟體與協定應用上，包含 **GPU Direct RDMA**（GPU 記憶體直接互傳）、**NVMe over Fabrics**（高速儲存），以及為了向下相容傳統 IP 網路的 **IPoIB (IP over InfiniBand)**。

## Physical Connectivity

### Host Channel Adapters 與 ConnectX

簡單來說，**Host Channel Adapters (HCA)** 是一個技術名詞，指的是在 InfiniBand 或高效能網路架構中，安裝於伺服器端並提供運算節點網路連接能力的介面卡，其功能等同於傳統乙太網路環境中的網路卡 (NIC) 。而 **ConnectX** 則是 NVIDIA（原 Mellanox Technologies）所開發的 HCA 家族產品系列名稱。因此，兩者的關係是技術類別與具體產品的關係。

#### 分層拆解：HCA 與 ConnectX 架構

* **硬體層 (實體規格)：** ConnectX 是一系列具備統一架構的 HCA，單張網卡即可同時支援 InfiniBand 與 Ethernet 網路連線 。早期的 ConnectX-1 型號僅提供基本的網路連線能力，但現代的型號（如 ConnectX-8、ConnectX-9）已演進為具備高度可程式化與大量卸載 (Offload-heavy) 能力的硬體加速器 。此外，ConnectX HCA 的硬體架構同時也是建構 NVIDIA BlueField DPU 的底層核心基礎 。
* **傳輸層 (InfiniBand/RoCE 協定)：** 在傳輸層面，HCA 是執行 RDMA (Remote Direct Memory Access) 協定的關鍵硬體。透過 ConnectX，節點間的資料傳輸能完全繞過主機 CPU 與作業系統核心，達成極低負載的端到端資料轉發。這使得網路協定處理的工作從 CPU 卸載至 HCA 網卡硬體上，大幅提升效能並降低延遲。
* **管理層 (遙測與自動化)：** 在現代資料中心中，ConnectX HCA 可透過 DOCA 軟體框架或 UFM (Unified Fabric Manager) 進行深度的管理與監控。維運人員能結合 NVIDIA NetQ 或 UFM 的遙測技術，即時收集 HCA 端口的封包計數、壅塞控制狀態與健康度，達成網路的自動化維運與最佳化。

#### 軟硬體間的連動關係

ConnectX 展現了極高的軟硬體互操作彈性。在硬體端，許多 ConnectX 網卡型號支援 VPI (Virtual Protocol Interconnect) 技術，這表示同一張 HCA 網卡能透過軟體切換，同時相容於 InfiniBand 網路與 Ethernet (包含 RoCE) 架構。在軟體端，它能與上層的 AI 通訊函式庫（如 NVIDIA NCCL）深度連動。當 NCCL 發起分散式運算請求時，會直接調用 ConnectX 內的 RDMA 引擎與硬體佇列，讓軟體定義的通訊邏輯與 HCA 晶片的線速轉發完美結合。

## Cables

### InfiniBand 線材與網路架構

#### 硬體層 (實體規格)

在 InfiniBand（如 NDR 400Gb/s）架構中，主要使用 OSFP 或 QSFP 規格的連接器，依照傳輸介質與耗能特性，線材主要分為三大類與其對應的使用時機：

* **被動銅纜 (Passive Copper Cables / DAC)：**
  * **特性：** 線材重量較重且缺乏彈性。由於內部沒有主動的訊號放大晶片，因此功耗極低（以 QM9700 為例，使用被動銅纜的典型系統功耗約為 747W）。
  * **使用時機：** 適用於**機櫃內 (Intra-rack)** 的極短距離連線，通常用於伺服器 HCA 連接至同一機櫃的機頂交換器 (ToR)，長度限制通常在 2 公尺以內。
* **主動光纖纜線 / 光學收發器 (Active Optical Cables (AOC) / Optical Modules)：**
  * **特性：** 包含多模/單模光纖 (SMF) 及高功率收發器（如 LR4）。兩端接頭內建光電轉換晶片，因此會顯著增加系統功耗（以 QM9700 為例，滿載主動線材的最大功耗會飆升至 1,720W）。
  * **使用時機：** 適用於**跨機櫃 (Inter-rack)** 的長距離傳輸。用於連接 Leaf 交換器與 Spine 交換器，或是跨資料中心樓層的連線（例如大於 30 公尺的場景）。
* **分支線材 (Splitter / Breakout Cables)：**
  * **特性：** 採用 1:2 的分支設計，能將單一實體的四通道 (Quad-lane) 連接埠（如 400G OSFP），在物理上分割為兩個獨立的雙通道 (Dual-lane) 連接埠（如兩個 200G NDR200）。
  * **使用時機：** 當需要**最大化交換器埠密度**以連接更多運算節點時使用。此線材可將 64 埠的 NDR 400G 交換器轉換為 128 埠的 NDR200 200G 機頂交換器。

#### 傳輸層 (InfiniBand 錯誤修正與協定)

在傳輸層面，實體線材的長度與類型會直接決定底層除錯協定的啟動條件。為了確保零掉包（Lossless），當被動銅纜長度超過 2 公尺，或是主動光纖長度超過 30 公尺時，系統必須啟用低延遲的里德-所羅門前向錯誤更正 (LL RS FEC) 機制，以修復傳輸過程中的訊號衰減。此外，若光纖線長超過 30 公尺，連結層重傳 (LLR, Link Level Retransmission) 功能將因為物理延遲過高而無法運作。

#### 管理層 (遙測與自動化)

透過 MLNX-OS 與底層韌體，維運團隊可對線材進行極為深度的遙測。管理層會透過 I2C 讀取收發器 EEPROM 中的資料（如製造商、料號、溫度、電壓、Tx/Rx 傳輸功率等）。維運系統能持續監控線材的光學訊號雜訊比 (SNR) 與原始位元錯誤率 (Raw BER)。如果偵測到非 NVIDIA 認證的線材，或線材合規碼 (Compliance code) 不符，系統甚至會在管理層觸發警報並強制關閉該連接埠（例如發出 `Port is closed due to cable stamping` 錯誤）。

#### 解決大型 AI 叢集的瓶頸

在建構如 DGX SuperPOD 級別的大型 AI 叢集時，網路成本與實體空間是極大的瓶頸。透過妥善使用**分支線材 (Breakout Cables)**，架構師能夠在不增加核心交換器數量的前提下，提供雙倍的存取層 (Access Layer) 節點數量。這種設計不僅減少了實體佈線的空間需求與功耗，更使得中小規模的 AI 叢集能夠以兩層式的胖樹 (Fat Tree) 拓撲架構進行橫向擴展，確保 GPU 在處理龐大參數同步（大象流）時，能享有極致的吞吐量與低延遲。

#### InfiniBand 與線材

分支線材的使用展現了高度的軟硬體互操作性。在硬體端插入 Splitter Cable 後，架構師必須在 MLNX-OS 中將系統設定檔更改為 `ib split-ready`，接著透過軟體 API 將連接埠模組類型更改為 `qsfp-split-2`。此時，底層的 Quantum ASIC 晶片會進行硬體資源的重新分配，將 1 個實體埠邏輯拆分為 2 個獨立的子介面（命名規則由 `x/y` 變更為 `x/y/z`）。這種連動機制確保了上層 Subnet Manager 的路由計算能與底層被改變的物理線路完美對齊。

## InfiniBand Modules

場景： InfiniBand 連接 2 個 HCA 節點時，我可以如何驗證這些裝置和連線狀態 ?

### InfiniBand 架構與驗證流程

在 InfiniBand 網路中，所謂的*InfiniBand Modules*主要指構建 RDMA 傳輸與底層硬體溝通的核心軟體驅動模組。當連接兩個 HCA 節點時，架構可拆解如下：

1. **硬體層 (實體規格與連線)：**
    在實體層面，兩個節點需要安裝 ConnectX HCA 網卡，並透過被動銅纜 (DAC) 或光纖線材與交換器（或兩節點直連）對接。硬體層的最基本驗證方式是觀察網卡上的 Link LEDs 燈號是否亮起（例如顯示為 Green/Solid amber），這代表實體層 (Physical state) 已成功連線。
2. **傳輸層 (InfiniBand Modules 軟體協定堆疊)：**
    在作業系統內部，必須載入 OpenFabrics Enterprise Distribution (OFED) 驅動程式，這些核心的 InfiniBand Modules 包含：
    * **`ib_uverbs`**：提供使用者空間 (User-space) 的 RDMA API，讓上層應用（如 MPI 或是 NVIDIA NCCL）可以直接呼叫。
    * **`ib_core`**：Linux 核心中的 RDMA 子系統，負責提供 verbs API、設備抽象化、佇列對 (Queue Pair) 處理與記憶體註冊。
    * **`mlx4_ib` / `mlx5_core`**：底層硬體驅動程式，負責管理 PCIe 設備、中斷處理，並將硬體與 RDMA 軟體堆疊橋接起來。
3. **管理層 (Subnet Manager 與遙測自動化)：**
    InfiniBand 網路沒有傳統的 ARP 機制，因此兩個節點要能互通，**必須在其中一個節點上執行 Subnet Manager (SM，例如 `opensm`)**。SM 是整個 Fabric 的大腦，它會掃描拓樸，並依據硬體燒錄的 64-bit GUID 分配 16-bit 的 LID (Local Identifier) 給每個連接埠，以建立硬體轉發的路由表。

### 如何確認裝置與連線狀態

當完成兩個 HCA 節點的實體連接與 OFED 軟體模組安裝後，可利用以下標準的 InfiniBand 診斷工具 (InfiniBand Diags) 進行驗證：

* **裝置與硬體狀態驗證 (`ibv_devinfo` / `ibstat`)：**
    執行 `ibv_devinfo` 可驗證作業系統是否已正確載入 HCA 設備與驅動模組。執行 `ibstat` 則可深入查看連接埠狀態，需要確認輸出中的 `State` 為 `Active`、`Physical state` 為 `LinkUp`，並確認該連接埠已成功從 SM 獲取 `Base lid` 與 `SM lid`。
* **端到端連線驗證 (`ibping`)：**
    這是一個運作於 InfiniBand 層（基於 LID/GUID）的測試工具。可以在其中一個節點啟動伺服器模式執行 `ibping -S`，然後在另一台節點執行 `ibping -L <目標 LID>` 來發送 ping 請求，藉此驗證兩節點間的 InfiniBand 協定是否暢通。
* **整體拓樸驗證 (`ibnetdiscover` / `ibhosts`)：**
    在管理節點執行 `ibnetdiscover` 可掃描並繪製出整個 IB 網路的拓樸圖（包含 HCA、交換器與線路連接關係）。您也可以使用 `ibhosts` 來列出 Fabric 中所有成功被探索到的主機節點。

### 核心價值

在大型 AI 模型訓練與高效能運算 (HPC) 叢集中，節點（如 GPU）之間需要頻繁同步龐大的神經網路參數（大象流）。若使用傳統網路，封包處理會嚴重消耗 CPU 資源。InfiniBand 透過上述的 `ib_uverbs` 與 `ib_core` 模組，結合 GPUDirect RDMA 技術，讓 GPU 記憶體能直接與 HCA 網卡進行資料交換，完全繞過宿主機的 CPU 與作業系統核心。這種將網路基礎設施徹底卸載至硬體的設計，將端到端通訊延遲降至極低的 1~2 微秒 (µs)，從而解決了 GPU 等待網路 I/O 的效能瓶頸。

### InfiniBand 與 ConnectX

InfiniBand 的高效能來自軟硬體的深度互操作性。當上層的 AI 通訊函式庫（例如 NVIDIA NCCL）發起分散式訓練時，它會透過 `ib_uverbs` 模組，直接與底層的 ConnectX ASIC 硬體佇列連動。同時，硬體上的 GUID（全球唯一識別碼）會與管理層的 Subnet Manager (SM) 連動。SM 透過收集 GUID 並分配 LID，再將路由規則下發至各交換器的 LFT (Linear Forwarding Table) 中。這種由軟體（SM）集中控制、並由硬體（HCA 與 Switch ASIC）直接執行線速轉發 (Wire-speed forwarding) 的機制，達成了無損且無迴圈的高效網路。
