# Anycast 技術概觀與原理

* **定義與本質**：Anycast 並非一種新的協定，而是利用 Layer 3 網路固有的 **ECMP（等價多路徑）** 功能來提供「無狀態」的負載分擔服務。
* **運作方式**：
  * 透過 **Routing on the Host** 技術，在伺服器上直接執行 OSPF 或 BGP。
  * 多台伺服器同時宣告相同的服務 IP 位址（例如 `172.16.255.66/32`）。
  * 網路設備會根據硬體雜湊（Hardware Hash）L3 與 L4 標頭（Header）來決定流量路徑。
* **流量一致性**：對於同一個流量（Flow），雜湊結果會完全相同，確保該流量的所有封包都會送往同一個目的地。

### Anycast 與傳輸協定（TCP/UDP）的相容性

| 特性 | UDP 應用（如 DNS, NTP） | TCP 應用（如 HTTP, FTP） |
| :--- | :--- | :--- |
| **適用性** | **極度推薦**。 | **需謹慎使用**，建議僅限短連線。 |
| **故障影響** | 封包遺失通常由應用層重試解決，影響極小 。 | 網路變動可能導致流量偏移，造成連線中斷或重設（TCP Reset）。 |
| **狀態維護** | 無狀態（Stateless），不需維持連線資訊。 | 有狀態（Stateful），必須在同一個伺服器完成三向交握。 |


### 網路變動引發的「重新雜湊」（Rehashing）問題

這是在 Anycast 設計中最需要注意的技術風險：

1. **雜湊原理**：硬體雜湊函數會參考「可用的 Next hop 列表數量」進行 **Modulo（取模）** 運算 。
2. **變動風險**：當網路中增加或減少一個節點（Next hop）時，Modulo 數值會改變，導致**所有流量**（包含未受故障影響的正常流）可能被重新分配到不同的伺服器。
3. **結果**：如果 TCP 流量被導向一台沒有該連線紀錄的伺服器，伺服器會回傳 **TCP Reset**，導致該連線立刻斷線（例如下載中斷） 。

#### 彈性雜湊（Resilient Hashing）的侷限

* **功能**：可以防止因節點「故障/移除」而影響到與該節點無關的現有流量。
* **限制**：無法防止因「新增節點」而引發的重新雜湊現象。

### Anycast 部署建議與限制

* **建議場景**：
  * **無狀態應用**：如 DNS、NTP 或可於應用層共享狀態的服務。
  * **短連線 TCP**：如網頁瀏覽，即使中斷，使用者透過重新整理即可恢復。
* **應避免的場景**：
  * **長連線應用**：如 FTP 大檔案傳輸、影音串流。
  * **關鍵交易**：需要完整日誌紀錄與完成性的金融交易。
  * **多連線應用**：如 FTP（控制與數據連線獨立），雜湊可能將兩個連線導向不同伺服器而導致失敗。
* **流量發起方向**：不要從 Anycast IP 發起「主動對外」的 TCP 連線，因為回程流量可能不會回到原始發起的伺服器。


在 **Cumulus Linux** 交換器上，你可以使用以下指令來觀察 Anycast 的路由與雜湊路徑：

* 查看 Anycast 路由的多個 Next hop：`ip route show [Anycast_IP]` 。
* 預測硬體雜湊結果：使用 `cl-ecmpcalc` 指令，輸入來源/目的地 IP 與埠號來確認流量會走哪一個埠口。

**總結**：Anycast 是一種低成本、高擴充性的方案，但其成敗關鍵在於**應用層是否具備處理網路抖動（Flapping）的能力**。
